# YandexDirect-QualityUsers

## Описание

**YandexDirect-QualityUsers** — JavaScript-скрипт для отслеживания качественного, вовлечённого трафика в Яндекс Метрике. Когда посетитель проводит на сайте достаточно времени **и** взаимодействует с ним (скроллит, двигает мышью, выделяет текст), скрипт отправляет в Метрику цель `reachGoal`. Эту цель можно использовать для обучения автостратегий Яндекс Директа на реальных, заинтересованных пользователях, а не на «случайных» визитах и тривиальных ботах.

## Что отслеживается

Скрипт собирает поведенческие сигналы и вычисляет **скор от 0 до 100**. Цель срабатывает, когда:

1. **Минимальный гейт** пройден — посетитель был на странице хотя бы половину `MIN_TIME_ON_SITE` (т. е. ~15 секунд видимого времени) и активен хотя бы половину `MIN_ACTIVE_TIME` (~7,5 секунд), И был хотя бы один сигнал живой активности (скролл/мышь/выделение/копирование/devicemotion).
2. **Скор ≥ `GOAL_SCORE_THRESHOLD`** (по умолчанию 60).
3. Не превышен дневной лимит и не определён бот.

### Как считается скор

| Сигнал | Вклад | Насыщение |
|---|---|---|
| Видимое время вкладки | до 25 | 60 сек в foreground |
| Активное время (без idle) | до 25 | 30 сек реальной активности |
| Прочитанные секции | до 20 | 100% контентных секций (`<p>`/`<h2>`/`<h3>`/`<li>` в `<main>`/`<article>`) набрали dwell ≥ `max(800мс, слова × 200мс)` каждая. Если контентных секций <5 — fallback на глобальную глубину скролла |
| Выделил **или** скопировал текст | +10 / +15 | берётся **максимум**, не сумма (копирование почти всегда сопровождается выделением, складывать = двойной счёт за одно действие) |
| Возвращающийся (`visitCount ≥ 2`) | +5 | один раз |
| Постоянный (`visitCount ≥ 5`) | +5 | один раз |
| Внутренняя навигация ≥ 1 | +10 | клик по ссылке внутри сайта; накапливается через `localStorage` между MPA-страницами |
| Активный навигатор ≥ 3 | +5 | дополнительно за 3+ переходов |
| Штраф за «полёт без чтения» | −10 | если >70% пути проеханы быстрее 3000 px/сек при общем пути > 1000 px. Один трекпадный флик не штрафуется — учитывается **доля**, а не отдельные сэмплы |

Слабые сигналы складываются: например, юзер с 25 секундами видимого времени, 80% глубины скролла и выделением текста наберёт ~10 + 13 + 16 + 10 = 49 — недостаточно для срабатывания. А юзер, прочитавший статью 60 секунд активно, скрольнувший до конца и скопировавший фрагмент, наберёт 25 + 25 + 20 + 15 = 85 — цель срабатывает.

`GOAL_SCORE_THRESHOLD = 60` — это стартовая калибровка. Поправьте под свой сайт, посмотрев распределение параметра `score` в отчёте Метрики через 1–2 недели после установки.

## Параметры цели

При срабатывании цель отправляется в Метрику двумя вызовами:

**1) `reachGoal` — параметры цели** (видны в отчёте «Конверсии», для сегментации):

| Параметр | Тип | Что значит |
|---|---|---|
| `score` | 0–100 | Итоговый скор, по которому пробит порог |
| `visible_time_sec` | int | Секунды в foreground |
| `active_time_sec` | int | Секунды реальной активности |
| `scroll_depth_pct` | 0–100 | Максимальная глобальная глубина скролла, % |
| `reading_pct` | 0–100 | % прочитанных контентных секций (или fallback на глубину) |
| `visit_count` | int | Номер визита (≥2 — возвращающийся) |
| `text_selected` | 0/1 | Выделял текст |
| `copied` | 0/1 | Копировал текст |
| `internal_nav` | int | Кликов по внутренним ссылкам в этой и предыдущей (≤5 мин) странице |
| `exit_intent` | 0/1 | Курсор уходил за верхнюю границу окна (целился в адресную строку) |
| `fast_scroll_pen` | int | Применённый штраф за «полёт без чтения» (0 или 10) |

**2) `userParams` — параметры пользователя** (привязаны к ClientID, **доступны для Аудиторий Яндекса**):

| Параметр | Что значит |
|---|---|
| `quality_score_last` | Скор последнего срабатывания цели — для диагностики |
| `quality_score_max` | **Максимальный** достигнутый скор за всё время (накопительно в `localStorage`) |
| `quality_visits_total` | **Сколько всего раз** этот браузер квалифицировался как качественный |
| `quality_internal_nav_last` | Снапшот: сколько внутренних переходов было в последней квалифицирующей сессии |
| `quality_internal_nav_max` | **Максимум** кликов по внутренним ссылкам в одной сессии за всё время — для look-alike сильнее снапшота |
| `quality_reading_pct_last` | % прочитанного контента в последней квалифицирующей сессии |

`userParams` хранит только последнее значение по ключу, поэтому в нём специально лежат **агрегаты за всё время**, а не сиюминутный снапшот. Юзер, который квалифицировался 7 раз с пиковым скором 88, — гораздо более сильный seed для look-alike, чем тот, кто квалифицировался один раз со скором 88 в данный момент.

`userParams` — это и есть тот канал, по которому данные попадают в построитель сегментов в Аудиториях Яндекса для look-alike в Директе. Параметры цели одни в этом не работают — без `userParams` их не видно при построении аудитории.

## Защита от накруток и ботов

Это **best-effort клиентская эвристика**, не security boundary. Мотивированный headless-бот с подменённым `navigator.webdriver` и `userAgent` пройдёт фильтр — против таких сценариев нужен серверный фингерпринтинг, отдельная задача.

Скрипт отсекает:

- Известные краулеры по `User-Agent` (Googlebot, YandexBot, Bingbot и т. п.).
- Headless-Chrome (`HeadlessChrome` в UA, `navigator.webdriver === true`).
- Скрытое top-level окно (`innerWidth/innerHeight === 0`).
- Боты с «ровным» поведением мыши и скролла (анализ дисперсии перемещений).
- Аномальную частоту дискретных событий (>600 событий/мин по `scroll`/`keydown`/`touchstart`).

Дополнительно действует ограничение:

- **Не более 3 срабатываний цели в сутки с одного браузера** (хранится в `localStorage`, сбрасывается на новой дате).
- **Не более 1 срабатывания за сессию вкладки** (сбрасывается при перезагрузке).

## Настройка времени на сайте (`MIN_TIME_ON_SITE`)

Зависит от типа сайта. Подбирайте порог под среднее время реального взаимодействия:

| Тип сайта | Рекомендуемый порог |
|---|---|
| Лендинг (продающая страница) | 20–30 сек |
| Интернет-магазин | 30–60 сек |
| Информационный сайт | 45–90 сек |
| Блог, новости | 60–120 сек |

Меняется в начале файла `QualityUser.js`:

```js
var MIN_TIME_ON_SITE = 30000; // мс
```

### Запись чисел в JavaScript

В исходном файле числа записаны привычно: `30000`, `15000`, `5 * 60 * 1000`. В **минифицированной** версии (`QualityUser.min.js`) terser сжимает их экспоненциальной записью — `3e4`, `15e3`, `3e5` и т. п. Это валидный JavaScript-литерал, движок интерпретирует обе записи абсолютно одинаково:

```js
3e4    === 30000   // 3 × 10⁴ = 30 000 (например, 30 секунд)
15e3   === 15000   // 1,5 × 10⁴ = 15 000 (15 секунд)
3e5    === 300000  // 3 × 10⁵ = 300 000 (5 минут)
18e5   === 1800000 // 1,8 × 10⁶ = 1 800 000 (30 минут)
```

При правке константы прямо в `QualityUser.min.js` можно использовать любую запись — все три варианта работают идентично:

```js
MIN_TIME_ON_SITE=60000     // обычная запись
MIN_TIME_ON_SITE=6e4       // экспоненциальная (60 сек)
MIN_TIME_ON_SITE=60*1000   // выражение — JS вычислит на старте
```

### Готовые пресеты под типичные сайты

Стартовые значения, которые потом стоит откалибровать по реальным данным в Метрике через 1–2 недели:

#### Дилерский сайт / автосалон / лендинг с каталогом моделей

Короткие карточки товаров без длинных описаний, цель — заявка/звонок. Решение принимается быстро, типичный паттерн «прокрутил главную → открыл карточку → посмотрел цену».

```js
var MIN_TIME_ON_SITE       = 25000;  // 25 сек: лендинг короткий, не нагружайте порогом
var MIN_ACTIVE_TIME        = 12000;  // 12 сек активности (~прокрутил главную + одна модель)
var IDLE_THRESHOLD         = 20000;  // 20 сек простоя — длинные тексты молча не читают
var GOAL_SCORE_THRESHOLD   = 45;     // ниже дефолта: reading-tracker почти не работает
                                      // (нет <article>), copy/select редки на лендинге
var MAX_GOAL_COUNT_PER_DAY = 5;      // покупатель авто заходит несколько раз в день
                                      // (утром глянул, вечером показал жене)
```

**Почему `GOAL_SCORE_THRESHOLD = 45` ниже дефолта 60:** на дилерском лендинге у заинтересованного посетителя типичный скор складывается из ~10 (видимое время) + ~10 (активное) + ~14 (глубина скролла) + ~10 (один internal-nav к карточке модели) ≈ 44. С порогом 60 такой посетитель не пройдёт, с 45 — пройдёт. Если копий/выделений мало, а текста на сайте мало — пороги нужно снижать.

#### Интернет-магазин (полноценный, с фильтрами и карточками товаров)

```js
var MIN_TIME_ON_SITE       = 45000;  // 45 сек: пользователь сравнивает товары
var MIN_ACTIVE_TIME        = 20000;  // 20 сек активности
var IDLE_THRESHOLD         = 30000;  // 30 сек — может изучать характеристики молча
var GOAL_SCORE_THRESHOLD   = 55;     // чуть ниже дефолта: глубина и navigation работают хорошо
var MAX_GOAL_COUNT_PER_DAY = 4;
```

#### Информационный сайт / блог / новости (длинные тексты)

```js
var MIN_TIME_ON_SITE       = 60000;  // 60 сек: реальное чтение
var MIN_ACTIVE_TIME        = 30000;  // 30 сек активности
var IDLE_THRESHOLD         = 45000;  // 45 сек — лонгрид молча читают долго
var GOAL_SCORE_THRESHOLD   = 60;     // дефолт работает: reading-tracker, copy, select все валидны
var MAX_GOAL_COUNT_PER_DAY = 3;
```

После недели данных смотрите распределение параметра `score` в Метрике (Отчёты → Конверсии → ваша цель → Группировка по параметру `score`). Если в распределении много значений `30–45` у внешне нормального поведения — снижайте порог. Если много `70+` — поднимайте, чтобы повысить «качество качественных».

## Установка через Менеджер тегов Яндекса

Скрипт рассчитан на установку через [Менеджер тегов Яндекса](https://tagmanager.yandex.ru/) (далее — МТЯ). Файл `QualityUser.js` уже содержит обёрточные теги `<script>...</script>` — он вставляется в поле HTML «как есть».

### Шаг 1. Подготовьте контейнер МТЯ

Если контейнера ещё нет:

1. Откройте [tagmanager.yandex.ru](https://tagmanager.yandex.ru/) и войдите под своим Яндекс ID.
2. Создайте контейнер для вашего сайта.
3. Установите код контейнера МТЯ на сайт согласно инструкции в интерфейсе (один раз — он уже включает в себя счётчик Метрики, если вы его настроили).

### Шаг 2. Подготовьте скрипт

1. Скачайте `QualityUser.js` из [этого репозитория](https://github.com/it-vbg/YandexDirect-QualityUsers/blob/main/QualityUser.js) (или нажмите *Raw* и сохраните файл).
2. Откройте файл в текстовом редакторе и поправьте две константы в начале:

   ```js
   var METRIKA_COUNTER_ID = 12345678;     // замените на ID вашего счётчика Метрики
   var METRIKA_TARGET = 'QualityUser';    // имя цели — должно совпадать с настройкой в Метрике
   ```

   `METRIKA_COUNTER_ID` берётся в Яндекс Метрике в настройках счётчика. `METRIKA_TARGET` — любое короткое латинское имя без пробелов; запомните его, оно понадобится в шаге 4.

3. При желании поправьте `MIN_TIME_ON_SITE` (см. таблицу выше).

### Шаг 3. Создайте тег в МТЯ

1. В контейнере МТЯ откройте раздел **Теги** → **Создать тег**.
2. Имя тега — например, `QualityUser`.
3. Тип тега — **Пользовательский HTML** (произвольный HTML/JavaScript).
4. В поле HTML вставьте **полное содержимое** файла `QualityUser.js`, **включая** обёрточные теги `<script>...</script>`.
5. Триггер — **Все страницы** (или собственный триггер, если хотите ограничить).
6. Сохраните тег.

### Шаг 4. Создайте цель в Яндекс Метрике

1. Откройте [Яндекс Метрику](https://metrika.yandex.ru/) → выберите счётчик → **Настройка** → **Цели**.
2. **Добавить цель**.
3. Название — например, `Микро: QualityUser` (для своих отчётов).
4. Тип условия — **JavaScript-событие**.
5. **Идентификатор цели** — ровно та же строка, что в `METRIKA_TARGET` (по умолчанию `QualityUser`). Регистр и пробелы значимы.
6. Сохраните цель.

### Шаг 5. Опубликуйте контейнер

В МТЯ нажмите **Опубликовать**. Через несколько минут после первого визита вовлечённого пользователя цель начнёт фиксироваться в Метрике.

### Шаг 6. Подключите цель к Яндекс Директу

В Яндекс Директе при настройке кампании выберите эту цель в качестве ключевой для автостратегии. Для устойчивого обучения автостратегии нужно ~10 конверсий по цели в неделю на кампанию — если трафика мало и `MAX_GOAL_COUNT_PER_DAY = 3` (значение в коде) ограничивает объём, поднимите его в `QualityUser.js`:

```js
var MAX_GOAL_COUNT_PER_DAY = 3;
```

## Проверка работы

1. Откройте свой сайт в обычной вкладке браузера.
2. Откройте DevTools (F12) → **Network** → фильтр `mc.yandex`.
3. Подвигайте мышью / поскролльте / выделите текст на странице, выждите `MIN_TIME_ON_SITE` секунд.
4. В Network должен появиться запрос вида `mc.yandex.ru/watch/<COUNTER_ID>` с параметром, содержащим имя вашей цели.
5. В Метрике в **Отчёты** → **Конверсии** цель появится с задержкой до 30 минут (агрегация Метрики).

Если цель не приходит — проверьте:

- Установлен ли счётчик Метрики на странице (объект `window.ym` должен существовать).
- Опубликован ли контейнер МТЯ.
- Нет ли блокировщиков (uBlock Origin режет `mc.yandex.ru`).
- Не сидите ли вы за `?_ym_debug=1` (это просто включит дебаг-лог Метрики в консоли — оттуда видно, доходит ли `reachGoal`).

| `METRIKA_TARGET` | `'QualityUser'` | Имя цели — должно совпадать с идентификатором в Метрике. |

| `IDLE_THRESHOLD` | `30000` | Через сколько мс простоя счётчик активности встаёт на паузу. |

| `LAST_VISIT_AT_KEY` | `'interested_user_last_visit_at'` | Ключ `localStorage` для timestamp прошлого визита. |
| `QUALITY_TOTAL_KEY` | `'interested_user_quality_total'` | Ключ `localStorage` для накопительного счётчика квалификаций. |
| `QUALITY_SCORE_MAX_KEY` | `'interested_user_quality_score_max'` | Ключ `localStorage` для максимального скора за всё время. |
| `QUALITY_INTERNAL_NAV_MAX_KEY` | `'interested_user_quality_internal_nav_max'` | Ключ `localStorage` для максимума кликов по внутренним ссылкам за визит. |
| `INTERNAL_NAV_KEY` | `'interested_user_internal_nav'` | Ключ `localStorage` для счётчика внутренних переходов с TTL 5 мин. |
| `INTERNAL_NAV_TTL_MS` | `300000` (5 мин) | TTL записи о предыдущей внутренней навигации. |

| `FAST_SCROLL_RATIO` | `0.7` | Какая доля проеханного пути должна быть быстрой, чтобы получить штраф. |
| `READING_MIN_SECTIONS` | `5` | Минимум контентных секций для включения reading-tracker; иначе fallback на глубину скролла. |

## Преимущества

- **Цель отражает вовлечённость, а не факт визита.** Время + активность — обязательный AND, тихие визиты не считаются.
- **Меньше шума для автостратегий.** Тривиальные краулеры и базовые headless-сценарии отсекаются.
- **Не нужен бэкенд.** Один файл, один тег, никаких зависимостей.

## Ограничения

- Эвристика работает в браузере — продвинутый бот её обойдёт. Для серьёзной защиты от накруток нужен серверный анализ.
- На iOS события `devicemotion`/`deviceorientation` требуют `requestPermission()` из user gesture — этот скрипт его не запрашивает, поэтому на iOS они не сработают (но `MIN_TIME_ON_SITE` + скролл/выделение — сработают).
- Дневной счётчик привязан к браузеру (`localStorage`), при очистке кэша обнуляется.
